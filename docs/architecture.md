# Architecture

## Why an OS for Intel SGX in Rust?

Intel SGX enclaves run in a constrained environment: no system calls, no
standard library, no dynamic memory management by default.  Traditional
approaches embed the application directly into the enclave code, tightly
coupling business logic with the SGX SDK.

Enclave OS (Mini) takes a different approach: it provides a **minimal OS
layer** that handles the low-level concerns — TLS termination, key management,
sealed storage, attestation — so that application logic can be deployed
**on top** of the OS without modifying the enclave binary.

### Why Rust?

| Concern | Rust advantage |
|---------|----------------|
| Memory safety | No buffer overflows, use-after-free, or data races — critical inside a TCB |
| No runtime | No garbage collector, no hidden allocations — predictable SGX memory usage |
| Zero-cost abstractions | Traits, generics, and iterators compile to the same code as hand-written C |
| Ecosystem | `rustls` (TLS), `ring` (crypto), `serde` (serialization) — all pure Rust, no C dependencies inside the enclave |
| Tooling | Cargo, clippy, miri — first-class testing and static analysis |

Using Rust eliminates entire classes of vulnerabilities from the Trusted
Computing Base.  The only C code in the enclave is the Intel SDK glue
(generated by `sgx_edger8r`) and a small `getrandom` shim.

## Apache Teaclave SGX SDK

[Apache Teaclave](https://teaclave.apache.org/) provides a Rust SDK for
Intel SGX that maps the `std` library to SGX-compatible implementations.
It supplies:

- **`sgx_tstd`** — a subset of Rust's standard library that works inside enclaves
- **`sgx_tseal`** — data sealing (MRENCLAVE / MRSIGNER policies)
- **`sgx_tse`** — report and quote generation for attestation
- **`sgx_types`** / **`sgx_urts`** — FFI bindings and untrusted runtime

### Our Fork

Enclave OS uses a [Privasys fork](https://github.com/privasys/teaclave-sgx-sdk)
of the Teaclave SGX SDK, maintained on the `main` branch.  The fork tracks
upstream with the following patches:

- **Rust nightly-2025-12-01** support (updated target JSON, sysroot build)
- **SGX SDK 2.25** compatibility
- Fixes for `target-pointer-width` and `target-c-int-width` types in the
  `x86_64-unknown-linux-sgx.json` target specification

We use only the minimum set of Teaclave crates:

| Crate | Side | Purpose |
|-------|------|---------|
| `sgx_tseal` | Enclave | MRENCLAVE-bound data sealing |
| `sgx_tse` | Enclave | SGX report creation for attestation |
| `sgx_types` | Host | SGX type definitions and constants |
| `sgx_urts` | Host | Enclave loading and ECALL dispatch |

All cryptographic operations (TLS, AEAD, signing, hashing) are handled by
`rustls` + `ring`, not by Teaclave's crypto modules.

## Composable Module Design

### Core OS vs. Modules

The enclave OS core provides only three primitives:

1. **RA-TLS ingress** — accept TLS 1.3 connections with SGX attestation
2. **HTTPS egress** — make outbound HTTPS requests from inside the enclave
3. **Sealed KV store** — encrypted key-value storage on the host

All business logic is implemented as **pluggable modules** that conform to
the `EnclaveModule` trait.  Each module is a separate Rust crate under
`crates/enclave-os-*`.

### The `EnclaveModule` Trait

```rust
pub trait EnclaveModule: Send + Sync {
    /// Human-readable module name (e.g. "wasm", "vault").
    fn name(&self) -> &str;

    /// Handle a client request.  Return `Some(response)` to claim it,
    /// or `None` to pass to the next module.
    fn handle(&self, req: &Request) -> Option<Response>;

    /// Config leaves contributed to the Merkle tree.
    fn config_leaves(&self) -> Vec<ConfigLeaf> { Vec::new() }

    /// Custom X.509 OID extensions for the enclave-wide RA-TLS certificate.
    fn custom_oids(&self) -> Vec<ModuleOid> { Vec::new() }

    /// Per-app identities for SNI-routed X.509 certificates.
    fn app_identities(&self) -> Vec<AppIdentity> { Vec::new() }
}
```

**Request dispatch** is sequential: the OS iterates over registered modules
and returns the first `Some(response)`.  If all modules return `None`, the
OS returns an error to the client.

### Module Crates

| Module | Crate | Description |
|--------|-------|-------------|
| **egress** | `crates/enclave-os-egress` | HTTPS egress; owns the root CA bundle |
| **kvstore** | `crates/enclave-os-kvstore` | Sealed AES-256-GCM key-value store |
| **wasm** | `crates/enclave-os-wasm` | Wasmtime WASM runtime with WASI + SDK |
| **vault** | `crates/enclave-os-vault` | JWT-authenticated (ES256) secret store |

### Composition Crates

Modules are assembled into the enclave via a **composition crate** — a thin
crate that depends on `enclave-os-enclave` and the desired modules, and
provides a custom `ecall_run` entry point that registers them.

```rust
// In your composition crate's lib.rs:
crate::modules::register_module(Box::new(WasmModule::new()));
crate::modules::register_module(Box::new(VaultModule::new()));
```

The CMake build system supports pointing at an external composition crate:

```bash
cmake -DENABLE_WASM=ON -DWASM_ENCLAVE_DIR=/path/to/composition/enclave ..
```

This design means you can create a custom enclave with exactly the modules
you need, without forking the OS.

See [wasm-app-example](https://github.com/Privasys/wasm-app-example) for a
complete composition crate example.

## Host-Enclave Communication

### SPSC Queue Architecture

All communication between the host (untrusted) and enclave (trusted) flows
through **shared-memory SPSC (Single Producer, Single Consumer) queues** with
a single OCALL for wake-up signalling.

There are **two independent channel pairs**:

| Channel | Direction | Purpose |
|---------|-----------|---------|
| **Data Channel** | host ↔ enclave | Raw TCP bytes. The host TCP Proxy writes inbound socket data and reads enclave TLS output. |
| **RPC Channel** | enclave ↔ host | Control-plane requests: KV reads/writes, HTTPS egress, etc. |

Separating data and control planes means network I/O never contends with
KV or egress RPC.

```
┌──────────────────────────────────────────────────────────────────┐
│                        Host (Untrusted)                          │
│                                                                  │
│  ┌──────────────┐   ┌─────────────┐   ┌────────────────┐         │
│  │ TCP Proxy    │   │ KV Backend  │   │ RPC Dispatcher │         │
│  │ (tcp_proxy/) │   │ (kvstore/)  │   │ (spin-polls    │         │
│  │ :443         │   │             │   │  enc_to_host)  │         │
│  └──────┬───────┘   └──────┬──────┘   └────────┬───────┘         │
│         │                  │                   │                 │
│         │                  └───────┬───────────┘                 │
│         │                          │                             │
│  ┌──────┴───────────┐   ┌──────────┴──────────────────────────┐  │
│  │  Data Channel    │   │  RPC Channel                        │  │
│  │  (TCP bytes)     │   │  (KV ops, egress, module dispatch)  │  │
│  │  host↔enc [SPSC] │   │  enc↔host [SPSC]                    │  │
│  └──────┬───────────┘   └──────────┬──────────────────────────┘  │
│         │                          │                             │
├─────────┼──────────────────────────┼─────────────────────────────┤
│         │                          │    SGX boundary             │
│         │                          │    (4 ECALLs + 1 OCALL)     │
│  ┌──────┴──────────┐   ┌───────────┴─────────────────────┐       │
│  │  TLS Engine     │   │  RPC Client                     │       │
│  │  (rustls)       │   │  (encodes req → enc_to_host,    │       │
│  │  TCP bytes in,  │   │   reads responses)              │       │
│  │  HTTP out       │   └───┬──────────────┬──────────────┘       │
│  └──────┬──────────┘       │              │                      │
│         │             ┌────┴──────┐  ┌────┴───────┐              │
│         │             │ HTTPS     │  │ Sealed KV  │              │
│         │             │ Egress    │  │ Store      │              │
│         │             │ (rustls)  │  │ (AES-GCM)  │              │
│         │             └───────────┘  └────────────┘              │
│  ┌──────┴───────────────────────────────────────────────┐        │
│  │         Pluggable Modules (EnclaveModule)            │        │
│  │  ┌────────┐  ┌────────┐  ┌──────┐  ┌──────────────┐  │        │
│  │  │ Egress │  │ WASM   │  │Vault │  │ HelloWorld   │  │        │
│  │  └────────┘  └────────┘  └──────┘  └──────────────┘  │        │
│  └──────────────────────────────────────────────────────┘        │
│                    Enclave (Trusted)                             │
└──────────────────────────────────────────────────────────────────┘
```

The host runs **two dedicated threads** alongside the enclave:

- **TCP Proxy** — non-blocking TCP listener that accepts connections, assigns
  `conn_id`s, and shuttles raw bytes to/from the enclave over the Data Channel.
  The enclave terminates TLS internally, so the host never sees plaintext.
- **RPC Dispatcher** — spin-polls the RPC Channel for enclave requests
  (KV reads/writes, HTTPS egress, etc.) and dispatches them to the appropriate
  host-side handler.

**Why SPSC queues instead of traditional OCALLs?**

Each OCALL costs ~5-10μs for the SGX context switch.  A traditional design
with individual OCALLs for `net_send`, `net_recv`, `kv_get`, `kv_put`, etc.
would require ~12 OCALLs per request cycle.  SPSC queues eliminate this:

- The enclave writes requests directly to host memory (no OCALL needed)
- Only a single `ocall_notify()` wakes the host dispatcher
- The host writes responses back to the queue
- The enclave polls the queue for responses

### Queue Design

| Property | Value |
|----------|-------|
| Type | Lock-free ring buffer |
| Synchronization | `AtomicU64` head/tail pointers |
| Padding | Cache-line (64-byte) to prevent false sharing |
| Capacity | Power-of-2 (default: 2 MiB per queue) |
| Message framing | `[u32 LE length][payload]` |
| Memory | Allocated in host memory; enclave accesses via raw pointers (`[user_check]` EDL) |

### RPC Protocol

Compact binary format over the SPSC queues:

| Field | Request (14 bytes) | Response (16 bytes) |
|-------|-------------------|---------------------|
| Correlation ID | u64 | u64 |
| Method / Status | u16 (method enum) | i32 (status code) |
| Payload length | u32 | u32 |
| Payload | variable | variable |

### EDL Interface

The entire host-enclave API is four ECALLs and one OCALL:

```c
trusted {
    public int ecall_init_channel(...);       // Set up RPC SPSC queue
    public int ecall_init_data_channel(...);  // Set up data SPSC queue
    public int ecall_run(...);                // Start the enclave OS
    public int ecall_shutdown();              // Graceful shutdown
};
untrusted {
    void ocall_notify();  // The ONLY OCALL — wake-up signal
};
```

## Sealed Configuration

On first run, the operator provides:

- **Intermediary CA** certificate + private key (ECDSA P-256)
- **Egress CA bundle** (optional — Mozilla root CAs for HTTPS egress)

The enclave:

1. Generates an **AES-256 master key** via RDRAND (first run only)
2. Packs everything into a `SealedConfig` structure (version 3)
3. Seals it using `sgx_tseal` with **MRENCLAVE policy**
4. Stores the opaque ciphertext on the host via the KV store

On subsequent restarts, the enclave automatically unseals the config.  If
new parameters are provided, they update the sealed config (e.g. CA rotation).

The sealed config contains:

| Field | Description |
|-------|-------------|
| `master_key` | AES-256 key (32 bytes) for KV store encryption |
| `ca_cert_der` | DER-encoded intermediary CA certificate |
| `ca_key_pkcs8` | PKCS#8-encoded CA private key (ECDSA P-256) |
| `module_data` | Per-module opaque blobs (BTreeMap) |

## Randomness

All randomness inside the enclave comes from Intel's hardware RNG via the
`RDRAND` instruction.  This is accessed through:

- `ring::rand::SystemRandom` — which calls `getrandom`
- A `getrandom` shim that routes to `sgx_read_rand` (Teaclave SDK)

No PRNG seeds, no `/dev/urandom`, no OCALLs for randomness.

## Design Decisions

1. **SPSC queues over traditional OCALLs** — eliminates ~12 context switches per request
2. **Lock-free design** — `AtomicU64` with cache-line padding, no mutexes in the hot path
3. **Custom binary RPC** — 14/16-byte headers, no protobuf overhead in the TCB
4. **rustls over OpenSSL** — pure Rust, no C dependencies, smaller TCB
5. **ring for crypto** — well-audited, SGX-compatible via Teaclave
6. **Deterministic key encryption** — HMAC-SHA256 for KV keys enables lookups without decrypting all keys
7. **Per-connection RA-TLS certificates** — fresh key pair + SGX quote per connection
8. **RocksDB host KV store** — opaque encrypted blobs, tuned for point lookups
9. **Crate-per-module** — independent compilation and testing, minimal core TCB

## Production Deployment

Because TLS terminates **inside** the enclave, any front-end load balancer
must operate at **Layer 4 (TCP passthrough)** — it must not terminate TLS
itself.  The load balancer forwards opaque TCP to the host's TCP Proxy port,
which relays into the enclave.

See [Layer-4 TCP Proxy Setup](layer4-proxy.md) for Caddy (caddy-l4) and
HAProxy configurations, including SNI-based routing for multiple enclave
instances.
